# 仿 Codex App 内部 MCP 与 Skill 平台化加载需求文档

## 1. 文档说明

### 1.1 文档目的

本文档用于描述仿 Codex App 中 Agent 使用的 MCP 与 Skill 从公司内部平台动态加载的 MVP 实现方案。

当前系统中，Agent 使用的 MCP 配置和 Skill 内容可能依赖服务器本地配置文件或本地目录。后续要求所有 Agent 可使用的 MCP 与 Skill 必须来自公司内部平台，不能由服务器本地静态配置决定。

本需求文档的目标是先实现一个 MVP 版本，通过模拟内部平台数据，快速跑通以下链路：

```text
内部平台返回 MCP 配置和 Skill 文件链接
        ↓
仿 Codex App 后端拉取配置
        ↓
解析 MCP 配置
        ↓
下载并解析 Skill Markdown 文件
        ↓
注册到 Agent 运行时
        ↓
Agent 使用内部平台下发的 MCP 与 Skill 执行任务
```

---

## 2. 背景说明

### 2.1 当前现状

当前已有一个仿 Codex App 的网页版本，具备 Agent 会话、任务执行、代码理解、文件处理等基础能力。

目前 Agent 使用 MCP 和 Skill 的方式可能存在以下问题：

1. MCP 配置依赖服务器本地配置。
2. Skill 内容依赖服务器本地文件。
3. 不便于公司统一管理 Agent 可用能力。
4. 不便于后续做权限控制、版本控制、灰度发布和审计。
5. 不利于多个环境、多个业务场景复用统一 MCP 与 Skill 能力。

### 2.2 目标变化

后续 Agent 所使用的 MCP 和 Skill 必须来自公司内部平台。

其中：

```text
MCP：通过接口获取配置，配置格式与常见 MCP 配置基本一致。
Skill：通过接口获取 Skill 文件链接，后端下载文件后解析为标准 Markdown 内容。
```

MVP 阶段可以不接真实平台，先使用 Mock 数据模拟内部平台返回结果。

---

## 3. 需求目标

### 3.1 核心目标

实现 Agent 能力来源的平台化加载。

即：

```text
Agent 不再直接读取服务器本地 MCP / Skill 配置，
而是在会话创建或任务启动时，
从公司内部平台接口获取 MCP 与 Skill 配置。
```

### 3.2 MVP 目标

MVP 版本需要完成以下目标：

1. 新增内部平台资产获取接口，可以先使用 Mock 接口。
2. 后端支持拉取 MCP 配置。
3. 后端支持解析 MCP 配置。
4. 后端支持根据 Skill 文件链接下载 Markdown 文件。
5. 后端支持解析 Skill Markdown 内容。
6. 后端支持将 MCP 工具和 Skill 内容注册到 Agent 运行时。
7. Agent 回复时能够体现 Skill 已经生效。
8. 前端或调试接口可以查看当前 Agent 会话加载了哪些 MCP 和 Skill。
9. 任务执行过程中使用固定的 MCP 和 Skill 快照，避免运行中配置变化影响正在执行的任务。

### 3.3 非目标

MVP 阶段暂不实现以下能力：

1. 不实现完整权限系统。
2. 不实现复杂的 MCP 工具调用审批。
3. 不实现 Skill 市场或 Skill 管理后台。
4. 不实现 Skill 多文件包解析。
5. 不实现 Skill 自动路由和向量召回。
6. 不实现复杂的灰度发布。
7. 不实现完整审计系统。
8. 不实现 MCP Server 沙箱隔离。
9. 不实现危险命令拦截。
10. 不实现企业级密钥托管。

这些能力放到后续版本迭代。

## 4. 核心概念

### 4.1 Agent Asset

Agent Asset 表示 Agent 启动时需要加载的能力资产，包括：

```text
MCP 配置
Skill 配置
Skill 文件链接
资产版本号
启用状态
```

### 4.2 MCP

MCP 是 Agent 可连接的外部工具、服务或能力集合。

MVP 中 MCP 配置从内部平台返回，格式类似：

```json
{
  "mcpServers": {
    "mock-terminal": {
      "command": "node",
      "args": ["./mock-mcp-server/index.js"],
      "env": {
        "NODE_ENV": "development"
      }
    }
  }
}
```

### 4.3 Skill

Skill 是 Agent 可使用的能力说明文件。

MVP 中 Skill 以 Markdown 文件形式存在，内部平台只返回文件链接，后端负责下载并解析。

示例：

```json
{
  "id": "project-review",
  "name": "老项目接手盘点 Skill",
  "version": "1.0.0",
  "file_url": "http://localhost:9000/skills/project-review.md",
  "enabled": true
}
```

### 4.4 Asset Snapshot

Asset Snapshot 表示某一次 Agent 会话加载到的 MCP 与 Skill 快照。

同一个 Agent 任务执行过程中，应固定使用同一个 Asset Snapshot。

这样可以避免任务执行到一半时，平台上的 MCP 或 Skill 发生变化，导致 Agent 行为不一致。

## 5. 用户场景

### 5.1 场景一：创建 Agent 会话时加载内部平台资产

用户在仿 Codex App 中创建一个新的 Agent 会话。

系统需要：

1. 根据当前应用、用户、场景请求内部平台。
2. 获取 MCP 配置。
3. 获取 Skill 文件列表。
4. 下载 Skill Markdown 文件。
5. 解析 MCP 和 Skill。
6. 生成当前会话的资产快照。
7. 启动 Agent 任务。

### 5.2 场景二：Agent 根据 Skill 执行任务

用户输入：

```text
帮我接手这个老项目，先给我一份盘点报告。
```

如果当前资产中存在“老项目接手盘点 Skill”，Agent 应该按照该 Skill 中定义的步骤执行，并输出结构化盘点报告。

### 5.3 场景三：Agent 使用 MCP 工具

用户输入：

```text
查看当前项目目录结构。
```

如果当前资产中存在文件系统类 MCP 工具，Agent 可以调用对应 MCP 工具获取目录信息。

MVP 阶段可以先用 mock MCP 工具验证链路，例如：

```text
echo 工具
list_files 工具
read_file 工具
```

### 5.4 场景四：查看当前会话加载了哪些能力

研发或调试人员可以通过接口查看当前 Agent 会话加载的 MCP 与 Skill。

例如：

```http
GET /api/agent/sessions/{sessionId}/assets
```

返回：

```json
{
  "session_id": "s_001",
  "asset_snapshot_id": "asset_001",
  "asset_version": "mock-v1",
  "mcp_servers": [
    {
      "name": "mock-tool",
      "status": "connected",
      "tools": ["echo"]
    }
  ],
  "skills": [
    {
      "id": "project-review",
      "name": "老项目接手盘点 Skill",
      "version": "1.0.0"
    }
  ]
}
```

## 6. 总体流程

### 6.1 新建会话流程

```text
用户创建 Agent 会话
        ↓
后端创建 session_id
        ↓
调用内部平台资产接口
        ↓
获取 MCP 配置和 Skill 文件链接
        ↓
解析 MCP 配置
        ↓
下载 Skill Markdown 文件
        ↓
解析 Skill 元信息和正文
        ↓
连接 MCP Server
        ↓
获取 MCP Tools 列表
        ↓
生成 Asset Snapshot
        ↓
绑定 session_id 和 asset_snapshot_id
        ↓
构建 Agent Runtime
        ↓
执行用户任务
```

### 6.2 继续旧会话流程

```text
用户继续旧会话
        ↓
根据 session_id 查询 asset_snapshot_id
        ↓
加载历史 Asset Snapshot
        ↓
恢复 MCP 与 Skill 运行时
        ↓
继续执行任务
```

原则：

```text
旧会话优先使用历史快照，不自动切换到平台最新配置。
```

### 6.3 配置更新流程

MVP 阶段可以简单处理：

```text
平台配置更新
        ↓
新建会话时拉取最新配置
        ↓
旧会话不受影响
```

后续版本再支持手动刷新当前会话资产。

## 7. 功能需求

## 7.1 内部平台资产获取

### 7.1.1 功能说明

系统需要支持从内部平台获取 Agent 所需的 MCP 和 Skill 配置。

MVP 阶段先使用 Mock 接口模拟。

### 7.1.2 请求接口

```http
GET /mock/internal-agent-assets?appCode=codex-web&scene=default
```

后续真实接口可替换为：

```http
GET /api/internal-agent-assets?appCode=codex-web&scene=default
```

### 7.1.3 请求参数

| 参数        | 类型     | 必填 | 说明                                      |
| --------- | ------ | -: | --------------------------------------- |
| appCode   | string |  是 | 应用编码，例如 codex-web                       |
| scene     | string |  是 | 场景编码，例如 default、project-review、code-dev |
| userId    | string |  否 | 用户 ID，MVP 可不传                           |
| projectId | string |  否 | 项目 ID，MVP 可不传                           |

### 7.1.4 返回示例

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "asset_version": "mock-v1",
    "mcp": {
      "mcpServers": {
        "mock-tool": {
          "command": "node",
          "args": ["./mock-mcp-server/index.js"],
          "env": {
            "NODE_ENV": "development"
          }
        }
      }
    },
    "skills": [
      {
        "id": "project-review",
        "name": "老项目接手盘点 Skill",
        "version": "1.0.0",
        "description": "用于分析老项目结构、功能、环境和运行方式",
        "file_url": "http://localhost:9000/skills/project-review.md",
        "enabled": true
      }
    ]
  }
}
```

### 7.1.5 处理规则

1. 当 `code != 0` 时，认为资产获取失败。
2. 当 `mcp` 为空时，Agent 可以不加载 MCP。
3. 当 `skills` 为空时，Agent 可以不加载 Skill。
4. 当某个 Skill 的 `enabled=false` 时，忽略该 Skill。
5. 当某个 Skill 下载失败时，MVP 阶段可以跳过该 Skill，但需要记录错误日志。
6. 当 MCP 连接失败时，MVP 阶段可以跳过该 MCP，但需要记录错误日志。
7. 如果所有 MCP 和 Skill 都加载失败，需要允许 Agent 继续运行，但需要在调试信息中标明资产加载异常。

## 7.2 MCP 配置加载

### 7.2.1 功能说明

系统需要解析内部平台返回的 MCP 配置，并连接对应 MCP Server。

### 7.2.2 MVP 支持范围

MVP 阶段优先支持 stdio 类型 MCP。

支持字段：

```json
{
  "command": "node",
  "args": ["./mock-mcp-server/index.js"],
  "env": {
    "NODE_ENV": "development"
  }
}
```

暂不强制支持：

```text
SSE MCP
Streamable HTTP MCP
OAuth 鉴权
远程 MCP
复杂 Header 配置
```

### 7.2.3 配置结构

```json
{
  "mcpServers": {
    "serverName": {
      "command": "node",
      "args": ["./server.js"],
      "env": {
        "KEY": "VALUE"
      }
    }
  }
}
```

### 7.2.4 处理流程

```text
读取 mcp.mcpServers
        ↓
遍历每个 serverName
        ↓
校验 command 是否存在
        ↓
启动 MCP Server 进程
        ↓
创建 MCP Client 连接
        ↓
调用 tools/list
        ↓
获取工具列表
        ↓
注册到 Agent Tool Registry
```

### 7.2.5 工具命名规则

为避免不同 MCP Server 中工具重名，Agent 内部注册工具时建议使用：

```text
{serverName}.{toolName}
```

例如：

```text
mock-tool.echo
filesystem.read_file
filesystem.list_directory
```

### 7.2.6 异常处理

| 异常              | 处理方式                                    |
| --------------- | --------------------------------------- |
| MCP 配置为空        | 跳过 MCP 加载                               |
| command 为空      | 跳过该 MCP Server                          |
| MCP Server 启动失败 | 标记该 Server 为 failed                     |
| tools/list 调用失败 | 标记该 Server 为 connected_but_tools_failed |
| 工具重名            | 使用 serverName.toolName 避免冲突             |
| MCP 调用超时        | 返回工具调用失败信息给 Agent                       |

## 7.3 Skill 文件下载

### 7.3.1 功能说明

系统需要根据内部平台返回的 `file_url` 下载 Skill Markdown 文件。

### 7.3.2 Skill 配置结构

```json
{
  "id": "project-review",
  "name": "老项目接手盘点 Skill",
  "version": "1.0.0",
  "description": "用于分析老项目结构、功能、环境和运行方式",
  "file_url": "http://localhost:9000/skills/project-review.md",
  "enabled": true
}
```

### 7.3.3 处理流程

```text
读取 skills 列表
        ↓
过滤 enabled=false
        ↓
校验 file_url
        ↓
下载 Markdown 文件
        ↓
解析 Markdown 内容
        ↓
注册到 Skill Registry
```

### 7.3.4 下载规则

1. 仅下载 `enabled=true` 的 Skill。
2. `file_url` 为空时跳过。
3. 下载超时时跳过。
4. 下载内容为空时跳过。
5. 下载失败需要记录错误日志。
6. MVP 阶段可以不做文件缓存，后续可增加缓存。

### 7.3.5 超时建议

| 项目              |  建议值 |
| --------------- | ---: |
| 单个 Skill 下载超时   |  5 秒 |
| 所有 Skill 总下载超时  | 15 秒 |
| 最大 Skill 文件大小   | 1 MB |
| 单次最多加载 Skill 数量 | 20 个 |

## 7.4 Skill Markdown 解析

### 7.4.1 功能说明

系统需要把下载到的 Markdown 文件解析为 Agent 可使用的 Skill 内容。

### 7.4.2 推荐 Skill 文件格式

```md
---
id: project-review
name: 老项目接手盘点 Skill
version: 1.0.0
description: 用于分析老项目结构、功能、环境和运行方式
when_to_use:
  - 用户需要接手一个老项目
  - 用户需要梳理项目结构、运行环境和业务功能
---

# 老项目接手盘点 Skill

## 目标

帮助用户快速理解一个已有项目，包括项目结构、核心功能、启动方式、环境依赖和业务流程。

## 执行步骤

1. 扫描项目结构。
2. 识别技术栈。
3. 梳理启动方式。
4. 梳理核心模块。
5. 输出接手盘点报告。
```

### 7.4.3 解析结果

```json
{
  "id": "project-review",
  "name": "老项目接手盘点 Skill",
  "version": "1.0.0",
  "description": "用于分析老项目结构、功能、环境和运行方式",
  "metadata": {
    "when_to_use": [
      "用户需要接手一个老项目",
      "用户需要梳理项目结构、运行环境和业务功能"
    ]
  },
  "content": "# 老项目接手盘点 Skill\n\n## 目标\n..."
}
```

### 7.4.4 解析规则

1. 如果存在 YAML Front Matter，则解析元信息。
2. 如果不存在 YAML Front Matter，则使用平台返回的 Skill 配置作为元信息。
3. Markdown 正文作为 Skill 内容。
4. Skill ID 优先使用平台返回的 `id`。
5. Skill 名称优先使用平台返回的 `name`。
6. Skill 版本优先使用平台返回的 `version`。
7. 如果 Markdown 内容为空，则该 Skill 无效。
8. 如果 YAML 解析失败，MVP 阶段不阻断加载，可以只使用 Markdown 正文。

## 7.5 Skill 注册

### 7.5.1 功能说明

解析后的 Skill 需要注册到当前 Agent 会话的运行时中。

### 7.5.2 注册内容

```json
{
  "id": "project-review",
  "name": "老项目接手盘点 Skill",
  "version": "1.0.0",
  "description": "用于分析老项目结构、功能、环境和运行方式",
  "content": "Markdown 正文内容"
}
```

### 7.5.3 MVP 注入策略

MVP 阶段采用简单策略：

```text
当前会话加载到的 Skill 全部注入 Agent 上下文。
```

后续版本再改为：

```text
根据用户问题选择最相关的 1~3 个 Skill 注入。
```

### 7.5.4 Agent Prompt 拼接方式

示例：

```text
你是公司内部仿 Codex App 的 Agent。

以下是公司内部平台下发的 Skill，你必须优先遵循这些 Skill 的执行规则。

【Skill: 老项目接手盘点 Skill】
版本：1.0.0
描述：用于分析老项目结构、功能、环境和运行方式

# 老项目接手盘点 Skill

## 目标
...

## 执行步骤
...

当前你可以使用以下 MCP 工具：

1. mock-tool.echo
   描述：返回输入内容
   输入参数：{ "text": "string" }

规则：
1. 只能使用当前会话加载到的 MCP 工具。
2. 不要假设存在未注册的 MCP 工具。
3. 当用户任务与某个 Skill 匹配时，优先按照该 Skill 的步骤执行。
4. 如果 Skill 与用户明确要求冲突，优先遵循用户当前明确要求。
```

## 7.6 Asset Snapshot 生成

### 7.6.1 功能说明

每次新建 Agent 会话时，需要将本次加载到的 MCP 和 Skill 生成资产快照。

### 7.6.2 快照内容

```json
{
  "asset_snapshot_id": "asset_20260706_001",
  "asset_version": "mock-v1",
  "app_code": "codex-web",
  "scene": "default",
  "mcp": {
    "servers": [
      {
        "name": "mock-tool",
        "status": "connected",
        "tools": ["echo"]
      }
    ]
  },
  "skills": [
    {
      "id": "project-review",
      "name": "老项目接手盘点 Skill",
      "version": "1.0.0",
      "status": "loaded"
    }
  ]
}
```

### 7.6.3 处理规则

1. 新建会话时生成新的 Asset Snapshot。
2. 当前会话执行期间固定使用该 Snapshot。
3. 继续旧会话时优先加载旧 Snapshot。
4. 平台配置更新不影响历史 Snapshot。
5. Snapshot 中需要保存原始配置，便于排查问题。

## 15. 异常处理需求

## 15.1 内部平台接口异常

| 场景               | 处理方式                  |
| ---------------- | --------------------- |
| 接口超时             | 使用空资产继续启动 Agent，并记录错误 |
| 接口 500           | 使用空资产继续启动 Agent，并记录错误 |
| 返回格式错误           | 使用空资产继续启动 Agent，并记录错误 |
| asset_version 为空 | 使用系统生成版本号             |
| mcp 字段为空         | 跳过 MCP 加载             |
| skills 字段为空      | 跳过 Skill 加载           |

## 15.2 Skill 异常

| 场景            | 处理方式        |
| ------------- | ----------- |
| file_url 为空   | 跳过该 Skill   |
| 下载超时          | 跳过该 Skill   |
| 文件不存在         | 跳过该 Skill   |
| Markdown 为空   | 跳过该 Skill   |
| YAML 解析失败     | 使用正文内容继续加载  |
| Skill 文件过大    | 跳过该 Skill   |
| 单个 Skill 加载失败 | 不影响其他 Skill |

## 15.3 MCP 异常

| 场景              | 处理方式            |
| --------------- | --------------- |
| command 为空      | 跳过该 MCP         |
| MCP Server 启动失败 | 标记 failed       |
| MCP 连接失败        | 标记 failed       |
| tools/list 失败   | 标记 tools_failed |
| tools/call 超时   | 返回工具调用失败        |
| 工具执行异常          | 返回异常信息          |
| MCP 进程退出        | 标记 disconnected |
